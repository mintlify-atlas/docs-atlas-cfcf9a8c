## Overview

The `dev` command provides utilities for Desloppify contributors and plugin developers. These tools help scaffold new language support, generate boilerplate, and maintain the codebase.

<Warning>
These are developer tools for **extending Desloppify itself**, not for using Desloppify in your projects. Most users won't need these commands.
</Warning>

## Usage

```bash
desloppify dev <action> [options]
```

## Subcommands

### scaffold-lang

Generate boilerplate for adding a new language plugin to Desloppify.

```bash
desloppify dev scaffold-lang <name> [options]
```

**Arguments:**

| Argument | Required | Description |
|----------|----------|-------------|
| `name` | Yes | Language name (lowercase, underscores allowed) |
| `--extension <ext>` | Yes | File extension(s) for the language (repeatable) |
| `--marker <marker>` | No | Generated file markers to detect (repeatable) |
| `--default-src <dir>` | No | Default source directory (default: `src`) |
| `--force` | No | Overwrite existing language directory |
| `--no-wire-pyproject` | No | Skip updating `pyproject.toml` |

**Examples:**

```bash
# Scaffold a Kotlin plugin
desloppify dev scaffold-lang kotlin --extension .kt --extension .kts

# Scaffold Go with custom source directory
desloppify dev scaffold-lang go --extension .go --default-src cmd

# Scaffold with generated file markers
desloppify dev scaffold-lang protobuf \
  --extension .proto \
  --marker "DO NOT EDIT" \
  --marker "Code generated by protoc"

# Overwrite existing language plugin
desloppify dev scaffold-lang python --extension .py --force
```

## What scaffold-lang creates

<AccordionGroup>
  <Accordion title="Directory structure">
    ```
    desloppify/languages/<name>/
    ├── __init__.py          # Package initialization
    ├── config.py            # Language configuration
    ├── detectors.py         # Custom detectors (stub)
    ├── fixers.py            # Auto-fixers (stub)
    ├── zones.py             # Zone classification rules
    └── tests/
        ├── __init__.py
        └── test_basic.py    # Basic tests
    ```
  </Accordion>

  <Accordion title="config.py">
    Defines the language configuration class inheriting from `LangConfig`:
    
    ```python
    class KotlinConfig(LangConfig):
        name = "kotlin"
        extensions = [".kt", ".kts"]
        default_src = "src"
        integration_depth = "standard"
        # ... more settings
    ```
  </Accordion>

  <Accordion title="zones.py">
    Zone classification rules for identifying production, test, config files:
    
    ```python
    ZONE_RULES = [
        ZoneRule("test", path_patterns=["**/test/**", "**/*Test.kt"]),
        ZoneRule("config", file_names=["build.gradle.kts"]),
        # ...
    ]
    ```
  </Accordion>

  <Accordion title="detectors.py & fixers.py">
    Stub files for implementing custom analysis and auto-fix logic:
    
    ```python
    # detectors.py
    def detect_issues(file_path, ast):
        """Implement language-specific detection."""
        return []
    
    # fixers.py
    def fix_issue(file_path, issue):
        """Implement auto-fix for detected issues."""
        pass
    ```
  </Accordion>

  <Accordion title="tests/">
    Basic test structure to verify the plugin loads:
    
    ```python
    def test_config_loads():
        from desloppify.languages.kotlin import config
        assert config.KotlinConfig.name == "kotlin"
    ```
  </Accordion>
</AccordionGroup>

## Language name validation

The language `name` must:
- Start with a letter
- Contain only lowercase letters, numbers, and underscores
- Not be a Python keyword (`if`, `for`, `class`, etc.)

**Valid names:**
- `python`
- `typescript`
- `rust`
- `objective_c`
- `c_sharp`

**Invalid names:**
- `TypeScript` (uppercase)
- `c++` (special characters)
- `class` (Python keyword)
- `2lang` (starts with number)

## File extension validation

Extensions:
- Must start with `.` (added automatically if omitted)
- Can only contain lowercase letters and numbers
- At least one extension is required

**Valid:**
- `.py`
- `.ts`
- `.js`
- `.cpp`

**Invalid:**
- `py` (missing dot - auto-corrected)
- `.PY` (uppercase - rejected)
- `.c++` (special chars - rejected)

## Pyproject.toml integration

By default, `scaffold-lang` updates `pyproject.toml` to include the new language's tests in pytest configuration:

```toml
[tool.pytest.ini_options]
testpaths = [
    "desloppify/languages/python/tests",
    "desloppify/languages/kotlin/tests",  # Added automatically
]
```

Skip this with `--no-wire-pyproject` if managing test paths manually.

## Development workflow

<Steps>
  <Step title="Scaffold the language">
    ```bash
    cd /path/to/desloppify
    desloppify dev scaffold-lang rust --extension .rs
    ```
  </Step>

  <Step title="Implement zone rules">
    Edit `desloppify/languages/rust/zones.py` to define patterns for test files, config files, etc.
  </Step>

  <Step title="Implement detectors">
    Edit `desloppify/languages/rust/detectors.py` to add language-specific code analysis.
  </Step>

  <Step title="Add tests">
    Create test files in `desloppify/languages/rust/tests/` to verify your implementation.
  </Step>

  <Step title="Run tests">
    ```bash
    pytest desloppify/languages/rust/tests
    ```
  </Step>

  <Step title="Register the language">
    The language is automatically discovered via `__init__.py`. No manual registration needed.
  </Step>

  <Step title="Test in a project">
    ```bash
    cd /path/to/rust/project
    desloppify scan
    ```
  </Step>
</Steps>

## Class naming

The config class name is auto-generated from the language name:

| Language name | Class name |
|---------------|------------|
| `python` | `PythonConfig` |
| `typescript` | `TypescriptConfig` |
| `objective_c` | `ObjectiveCConfig` |
| `rust` | `RustConfig` |

Pattern: Capitalize each word (split on `_`), append `Config`.

## Integration depths

The scaffolded config defaults to `integration_depth = "standard"`. Adjust based on your implementation:

| Depth | Description | What to implement |
|-------|-------------|-------------------|
| `minimal` | Basic file scanning | Zone rules only |
| `shallow` | Zone + simple checks | + Generic detectors |
| `standard` | Custom detectors | + Language-specific detectors |
| `full` | Deep analysis + fixes | + AST parsing, auto-fixers |

## Force overwrite

Using `--force` replaces all generated files, **losing any manual changes**. Use carefully:

```bash
# Backs up your work first
cp -r desloppify/languages/rust desloppify/languages/rust.backup

# Safe to regenerate now
desloppify dev scaffold-lang rust --extension .rs --force
```
